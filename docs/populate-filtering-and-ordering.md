# Advanced Filtering and Ordering for Related Entities

This document explains how to use advanced filtering and ordering features for related entities in queries using our repository abstraction layer.

## Overview

When working with related entities, it's often useful to filter or order the related entities independently of the main entity. MikroORM provides several options for this, which are now integrated into our repository layer:

- `populateWhere`: Filter which related entities are included using INNER JOIN semantics
- `populateFilter`: Filter which related entities are included using LEFT JOIN semantics
- `populateOrderBy`: Order related entities independently of the main entity

## Basic vs. Advanced Filtering

### Basic Filtering (Using dot notation in where clause)

When you want to filter the main entities based on properties of related entities:

```typescript
// Find users who have at least one completed order
const usersWithCompletedOrders = await userRepository.fetch({
  'orders.status': 'completed',
});
```

This returns only users who have at least one completed order.

### Advanced Filtering (Using populate options)

When you want all users, but only include specific related entities:

```typescript
// Find all users, but only populate their completed orders
const allUsersWithCompletedOrders = await userRepository.fetch(
  {}, // Empty where clause - get all users
  {
    populate: ['orders'],
    populateWhere: {
      orders: { status: 'completed' },
    },
  },
);
```

This returns all users, but the `orders` collection for each user only includes completed orders.

## populateWhere vs. populateFilter

### populateWhere

Uses INNER JOIN semantics when applied to nested conditions. If a related entity doesn't match the condition, it won't be included in the populated collection.

```typescript
// Get all users with only their high-value orders
const result = await userRepository.fetch(
  {},
  {
    populate: ['orders'],
    populateWhere: {
      orders: { amount: { $gt: 1000 } },
    },
  },
);
```

### populateFilter

Uses LEFT JOIN semantics. This is particularly useful for filtering in nested relationships without losing parent records.

```typescript
// Get all users with only their high-value orders
const result = await userRepository.fetch(
  {},
  {
    populate: ['orders'],
    populateFilter: {
      orders: { amount: { $gt: 1000 } },
    },
  },
);
```

The key difference is in how these behave with deeper nesting. For example, when dealing with `user -> orders -> items`:

- `populateWhere` with a condition on items will remove orders that don't have matching items
- `populateFilter` with a condition on items will keep all orders, but only populate the matching items

## populateOrderBy

Orders the related entities independently of the main entity.

```typescript
// Get all users with their orders sorted by date (most recent first)
const result = await userRepository.fetch(
  {},
  {
    populate: ['orders'],
    populateOrderBy: {
      orders: { createdAt: 'DESC' },
    },
  },
);
```

Each user's orders will be sorted by creation date in descending order.

## Complex Example

You can combine these features for complex querying:

```typescript
// Get users who have placed an order in the past month,
// populate only their completed orders,
// and sort those orders by amount descending
const usersWithRecentActivity = await userRepository.fetch(
  {
    // Main entity filter - users must have at least one recent order
    'orders.createdAt': { $gt: oneMonthAgo },
  },
  {
    // Main entity sorting - sort users by name
    orderBy: { name: 'ASC' },

    // Which related entities to include
    populate: ['orders', 'orders.items'],

    // Filter which orders to include in the populated collection
    populateWhere: {
      orders: { status: 'completed' },
    },

    // How to sort the orders in each user's collection
    populateOrderBy: {
      orders: { amount: 'DESC' },
    },
  },
);
```

## Performance Considerations

- Using populate features can impact performance, especially with deep nesting. Monitor query performance.
- Use appropriate indexes on fields used in `populateWhere` and `populateFilter` conditions.
- Consider using `fetchFields` or `fetchExclude` to limit the data returned when working with large entities.

## Debugging

To see the SQL queries generated by MikroORM when using these features, enable query logging:

```typescript
// In your application configuration
const config = {
  debug: true,
  logger: console.log,
};
```

This will log the generated SQL queries to the console, which can be useful for debugging and optimization.
